# 2026-07-02 — БАБХ ДНЕВНИК register + auto-generation (PR 3/3)

**Commit:** `feat(farm-record): auto-generate the ДНЕВНИК on completion + register + journal trigger`

## Design

Final PR of the roadmap. PR 1 captured the data, PR 2 built the generator + a
one-button trigger; this PR makes the ДНЕВНИК a first-class surface:

- **Auto-generation on completion.** A BullMQ job `farm-record-pdf` is enqueued
  (fire-and-forget, fail-open) when a `FIELD_OPERATION` task auto-resolves in
  `markOperationParcel`. The job regenerates the task location's **current-season**
  diary (season start → now, so the register always holds a complete current
  document, not a one-row fragment) and saves it via the PR-2 generator with
  `save:true` (FileRecord, domain `reports`, generated by system). Deduped per
  task via a deterministic `jobId = farm-record:${taskId}`, so repeated line
  transitions never double-generate. Task completion NEVER fails because PDF
  generation did — the enqueue is wrapped like audit streaming (fail-open).
- **Manual generation from the Journal entry.** Each completed line already
  mints an INPUT_APPLICATION LogEntry linked via `operationParcelId`. The journal
  detail surfaces `fieldOperation { taskId, locationId }`; when present, a
  "Дневник (PDF)" action opens a date-range modal (default current season) that
  streams the download. Hidden for entries with no operation linkage; the task
  page gets no button (the journal entry is the manual surface).
- **Register.** A "Farm records" tab on the Location detail lists the generated
  diaries (FileRecords, domain `reports`, filtered to this location), newest
  first with the latest marked "текущ"; each row downloads via the file route.
- **Season-wide.** A per-season row action generates one combined diary across
  every location with completed ops in the season (one section-set per location,
  page-break between).
- **Completeness affordances.** When the farm profile or operator certificates
  are missing, the generate modal shows a non-blocking `InlineNotice` linking to
  the admin pages — generation is never blocked (paper reality: blanks are filled
  by hand). The auto-generation job logs the same gaps as a warn.

## Decisions

- **Auto-gen regenerates the whole current-season diary**, not a per-task
  fragment — the register row is always a complete, current, printable document.
  `jobId = farm-record:${taskId}` dedups the burst of line-completion transitions
  into one generation.
- **Fail-open enqueue.** Completing a farm task must not fail because the queue
  is down or PDF generation threw — the enqueue is a try/catch warn after the
  tenant tx commits, mirroring the audit-stream fail-safe.
- **Worker needs the fonts.** The PDF is generated inside the BullMQ worker,
  which reads `src/lib/pdf/fonts/*.ttf` from `process.cwd()` — esbuild does NOT
  bundle `.ttf`. PR 2's Dockerfile COPY of `src/lib/pdf/fonts` lands them at
  `/app/src/lib/pdf/fonts`, which is the worker's cwd, so both the web route and
  the worker resolve them.
- **FileRecord discriminator.** Diaries are `domain:'reports'` with the filename
  `dnevnik-${locationId}-${from}_${to}.pdf`, so the register filters by location
  and parses the period from the name; `uploadedByUserId` distinguishes manual
  (a user) from auto (`null` → "автоматично").
- **E2E covers only the manual path** — CI runs no BullMQ worker, so the
  auto-generation enqueue is proven by a unit test on the resolution hook instead
  of end-to-end.
