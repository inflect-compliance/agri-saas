## Design

Before this change, rent/lease management was split across two surfaces and had
no tenant-wide home:

- **`RentRollCard`** тАФ a read-only summary (leased area, rent/season, expiring
  soon, CSV/PDF export) mounted per-location on the location Overview tab.
- **`ParcelLeasePanel`** тАФ the actual lease CRUD, reachable only from the parcel
  detail sheet on the map. A lease is parcel-bound, so it lived under the
  parcel-scoped API path.

This adds a dedicated top-level **Rent** page (`/t/:slug/rent`, in the *Govern*
sidebar section next to Trends) that folds both together:

- The tenant-wide **rent roll** summary (reuses `RentRollCard`, now tenant-wide
  when no `locationId` is passed).
- A **`DataTable` of every lease** across the tenant тАФ lessor ┬╖ parcel/location ┬╖
  type ┬╖ rent ┬╖ term ┬╖ status тАФ with inline edit + Epic-67 undo-toast delete.
- **Full create/edit** via a `Modal`. Because a lease is parcel-bound, create
  picks a parcel through a `Combobox` (`/leases/parcel-options`); the parcel is
  read-only on edit.
- Honors a `?locationId` deep-link. The location Overview tab's `RentRollCard`
  is **replaced** by a compact "View rent roll & leases" link into
  `/rent?locationId=тАж`.

### API shape

Leases were only reachable under the parcel path
(`тАж/locations/[id]/parcels/[parcelId]/leases`). To manage them tenant-wide the
register is **decoupled** into a flat namespace, reusing the existing usecases:

- `GET/POST /api/t/:slug/leases` тАФ list (`?locationId`), create (`parcelId` in
  the body via `TenantLeaseCreateSchema` instead of the path).
- `PATCH/DELETE /api/t/:slug/leases/[leaseId]`.
- `GET /api/t/:slug/leases/parcel-options` тАФ the create-modal picker.

The parcel-scoped routes still work identically тАФ the two just share the
underlying `create/update/deleteParcelLease` usecases.

## Decisions

- **Reuse the existing usecases, add only reads.** `create/update/delete` already
  took `parcelId`/`leaseId` as args, so the new flat routes are thin wrappers тАФ
  no duplicate write logic, no new audit events. Only two read functions were
  added.
- **Parcel picker over a locationтЖТparcel two-step.** A lease must attach to a
  parcel; a single searchable `Combobox` (grouped by location name via
  `optionDescription`) is cleaner than a cascading pair. Options are a lightweight
  id+name projection (`take:5000`), not the geometry-bearing row.
- **`remove` reads the cache via SWR's functional `mutate`, not a `leases`
  closure.** The column cells are memoised on `[t, tc, tl]`; snapshotting the
  outer `leases` variable would capture a stale array. SWR's `mutate` is
  referentially stable and its functional updater sees the current cache, so the
  optimistic remove + undo restore stay correct regardless of memoisation.
- **Index guardrail.** `listTenantLeases` is served by the existing
  `@@index([tenantId, endDate])` (the rent-roll expiry scan's index); the
  `createdAt` tiebreak sorts a tenant's bounded lease set in memory. The one new
  tenant-wide `Parcel` read (`listTenantParcelOptions`) is narrowed by the
  `tenantId`-leading `@@index([tenantId, locationId])`. Both reasons are recorded
  in `LIST_MODELS_TENANT_INDEX_SUFFICIENT`.
